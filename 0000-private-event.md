# Summary
[summary]: #summary

This RFC introduces the concept of **Private Chaincode Events** for Hyperledger Fabric. These events are generated by chaincode but differ from existing chaincode events in that they are stored and disseminated only to explicitly specified organizations and identities, without ever being recorded on the public ledger except the hash value. This feature enhances privacy and fine-grained control over event visibility.

# Motivation
[motivation]: #motivation

Hyperledger Fabric today only supports events that are visible to all users in the channel, i.e., public events. In this case, even the users employ private data for privacy protection, the corresponding events are still visible to parties outside of the private group.

Hence, numbers of Hyperledger Fabric users have expressed the need to emit chaincode events that are:

1. Invisible to unauthorized parties. Those events are only available for authorized users.
2. Not stored on the public ledger. This is to avoid private events being fetched by unauthorized users.
3. Delivered only to explicitly listed organizations and users.
4. Replayable from a specific block or timestamp.

Current public chaincode events cannot satisfy these privacy requirements. This proposal aims to provide a solution by building on existing private data collections (PDCs) and enhancing the event infrastructure to support controlled dissemination of private events.

# Guide-level explanation
[guide-level-explanation]: #guide-level-explanation

## Concept: Private Chaincode Events

Private Chaincode Events are triggered by invoking a new `SetPrivateEvent(name string, payload []byte) error` method in chaincode, which will insert the event data using the standard `PutPrivateData()` method, with a specially reserved prefixed key (e.g., `__hlf_privateEvent__eventName`). The special prefix is to easily distinguish from the user's stored private data.

The events can have the following structure by re-using PDC's member access control.

```json
{
  "payload": {
    "from": "A",
    "to": "B",
    "amount": 87
  }
}
```

There are requirements that implement more fine-granular access control on top of the PDC policies. In this case, we can use an extra `visibleTo` field, which allows to check the MSP and CN in the identity cert. In this case, the private events will conform to a structure as below:

```json
{
  "visibleTo": {
    "Org1MSP": ["user1"],
    "Org2MSP": ["user2", "user3"]
  },
  "payload": {
    "from": "A",
    "to": "B",
    "amount": 87
  }
}
```

This structure can help do the parsing and the potential extending in the future. The `visibleTo` field can help provide additional access control besides the PDC. If the user only needs to use PDC's policy, then this field can be set to empty. When the client certs got revoked and the peer will get the update and reject the client's access.

We do not want to propose another access control mechanism, but to try best to reuse the PDC policies.

The peer recognizes this structure as a private event and tracks it for notification purposes.


## Storing the Private Events

Since the private events are re-using the existing PDC storage, they do not need additional database. And note that the stored private events will use a special prefix as described in the concept.


## Listening to Private Events

Existing event listening API will be adopted, to minimize the change on client side.

The private events are restricted and stored in the similar way as the private data.

Only peers in organizations and CNs listed in the event's `visibleTo` map will process and relay the event to eligible subscribers.

When specific users subscribe to the private event, their identifies will be checked to match the `visibleTo` rules. Those unauthorized users will be rejected.

If a valid client's permissions change after events are stored, it will not be able to access the private events it can fetch before, as the PDC will restrict the access.

## Replaying Events

Private events are replay-able by leveraging their PDC storage. Even though they are not part of the public ledger, the peer can match them to their corresponding transaction ID and block, making replay feasible from a given block number.

The peer needs to change to identify the private event, read them from the PDC storage, and send to the client SDK.

## Purging Events
Since the private event belongs to the PDC, it can be purged as the PDC supports today. For example, if the PDC definition contains the `blockToLive` param, then the peer will periodically check and purge the outdated private data.

After the events are purges, the request to the specific events will get a nil/empty return.

# Reference-level explanation
[reference-level-explanation]: #reference-level-explanation

### Event Creation
- A chaincode uses `PutPrivateEvent(<pdc>, <name>, <eventData>)` to raise a private event. There is also an option to use the underlying PDC  API such as `PutPrivateData(<pdc>, "__hlf_privateEvent__<name>", <eventData>)`. This adds minimal effort for exsiting chaincode development.
- The `pdc` is the group of parties that can store the private events,`eventData` must be valid JSON with the `visibleTo` (optional) and the `payload` fields. 
- Peer intercepts and stores the event in the PDC and triggers event handling logic.

### Peer Notification Flow
1. Peer receives the request for creating a new `privateEvent` as chaincode invokes transaction happens.
2. Peer processes filtered blocks and detects private event keys, as well as the transactions are stored into the blocks.
3. Peer checks event visibility and constructs a payload with only allowed private events.
4. Client can fetch the event from valid peers using the existing events APIs. This requires minimal or no change of the client SDKs.

### Payload Format

The payload contains the detail of a transaction, e.g.,

```json
{
  "TransferDetails": {
    "from": "A",
    "to": "B",
    "amount": 100
  }
}
```

Payloads are Base64 encoded in the final event message.

### Replay Handling
- Peer tracks block and transaction IDs associated with private events.
- Replay request with block number or timestamp fetches eligible events from PDCs.
- Purging policies of PDCs may affect replay availability. It can eliminate the creation and storage of related private events.

### Other Considerations

In order to keep the compatibility with old version of peer nodes (those do not support private events), a new capability can be introduced. For a network mixed with peers of those two versions, the old peers can still support public events, and the new peers will support private event. If a client send the private event request to the old peers, it will be rejected with error message as the peer nodes can not recognize the request. And the chaincode invoke request to the old peer nodes will fail due to the lack of the new chaincode API `PutPrivateEvent`.

The upgrade of the peer nodes will follow today's upgrade workflow. Operators need to stop the peer node, replace the peer binary and restart. Finally, operators need to update channel configuration to the new capability (if need).

# Drawbacks
[drawbacks]: #drawbacks

- Adds complexity to peer event handling.
- Requires additional validation on event structure.
- Relies on PDC retention policies, which might limit replay.

# Rationale and alternatives
[alternatives]: #alternatives

**Alternatives considered:**

- Extend public chaincode event model to support visibility filters — rejected due to risk of leaking.
- Use external messaging service — need to integrate with fabric, adds latency and security risks.
- Let client SDK handle private event, instead of changing peer node - not preferred due to needing to modify existing application code, which is not scale considering there can be numerous running applications.
- Develop a standardized client-side library/wrapper that abstracts the PDC-based private event handling.

**Why this approach:**

There can be several approaches for various trade-off (client, peer, or others). The principle of this proposal is to keep the minimal change on client side to ease the application layer. As today's peer side already have good support of public events and PDC, it is a practical way to reuse these existing features as much as possible. From real user cases and feedbacks, this approach has the following advantages:

- Leverages existing private data collections, which guarantee the security preventing unauthorized access to private events.
- Keeps event flow within existing peer event architecture, with minimal (if not zero) changing effort on the client SDK.
- Fine-grained control over dissemination and visibility.
- The approach can coexist with the current public event system.

# Prior art
[prior-art]: #prior-art

Other distributed ledger systems such as Corda or Quorum support private messaging/events, but generally lack the granularity or replay capabilities integrated with block-level data. And they need an additional vault for each node. This RFC draws lessons from them while tailoring for Fabric's PDC model.

# Testing
[testing]: #testing

Test strategies include:

    - Subscriptions with/without valid CNs.
    - Replay logic from various block points.
- Regression tests ensuring public event behavior remains unchanged.

# Dependencies
[dependencies]: #dependencies

- Depends on enhancements to peer event infrastructure, including generating the private events, and allowing fetching private events from the delivering service.
- Related to proposals enhancing event replay mechanisms (if any).

# Unresolved questions
[unresolved]: #unresolved-questions

- Should the `visibleTo` structure support more complex identity attributes beyond CN? This is possible technically, however, we suggest using PDC's policy or CN as start.
- Should peers cache PDC private events for a configurable time to improve replay reliability? This is also a possible optimization for performance.
- Should private events update or deletion being allowed? We highly suggest not.
- How to prevent user's chaincode from overriding the private event in the private data? Unless private data introduces new write protection mechanism, otherwise, we cannot prevent the operation.
